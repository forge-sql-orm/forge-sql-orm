# Forge SQL ORM Example Scaffold

Bootstraps a new example Forge app under `examples/` using one of three templates. Each template mirrors the layout and conventions of the canonical examples in the repo.

## Inputs to collect

Before generating anything, confirm with the user:

1. **Example name** — used as the directory name. Must follow the pattern `forge-sql-orm-example-<slug>` (kebab-case). If the user gives only a slug, prepend the prefix.
2. **Template** — one of:
   - `basic-crud` — mirrors `examples/forge-sql-orm-example-simple` (single table, CRUD resolvers, migration V1).
   - `caching` — mirrors `examples/forge-sql-orm-example-cache` (two related tables with FK, KVS cache entity, `selectCacheable`, degradation queue).
   - `ai-vector` — mirrors `examples/forge-sql-orm-example-vector` (TiDB `vectorTiDBType` column, `vecCosineDistance` search resolver, seed data).
3. **Entity/table names** — at least one. For `caching` collect two (parent + child with FK). For `ai-vector` default to a single table with an embedding column, but let the user override the table name and vector dimension.
4. **Title** — the Jira global page title shown in `manifest.yml`.
5. **Include tests?** — default yes. Tests go under `__tests__/` inside the example directory and use Vitest with the root `vitest.config.mjs`.

If the user does not specify, default to `basic-crud` with a `users` table (id, name, email) and skip tests only if explicitly asked.

## What to generate

All paths are relative to `examples/<example-name>/`.

### Required files (all templates)

- `package.json` — copy structure from the template example. Keep the same `name` field convention (`jira-global-page-custom-ui`) unless the user objects. Scripts block must be identical to the template (`models:create`, `migration:create`, `migration:update`, `migration:drop`, `build:ci`, `forge:debug`, `build:husky`, `knip`). Use current pinned versions from the template's `package.json` — do not fetch latest; the root `autoupdate.sh` script owns version bumps.
- `manifest.yml` — generate a fresh Forge app id (`ari:cloud:ecosystem::app/<uuid>`) placeholder and tell the user to replace it with their own from `forge create`. Copy the `modules`, `sql`, `function`, `resources`, `app`, `permissions` sections from the template. Update the `title` to the user-provided value.
- `tsconfig.json` — identical to the template's (ESNext, strict, experimentalDecorators, etc.).
- `knip.config.ts` — identical to the template's (imports `defaultConfig` from `../../knip.config`).
- `README.md` — write from scratch with sections: title, one-paragraph description, Requirements, Quick Start (install / build frontend / register / deploy / install), Using the Application (list of resolvers), Notes. Mirror tone from the template's README.
- `src/utils/Constants.ts` — type definitions (response shapes, request payload types) derived from entities via `InferInsertModel` / `InferSelectModel` where natural.
- `src/entities/schema.ts` — Drizzle schema using `mysqlTable`, `primaryKey`, appropriate column types. Auto-increment int PK is the convention. For FKs use `.references(() => parent.id)` and add a matching `index()`.
- `src/entities/index.ts` — copy the exact auto-generated header comment (including `DO NOT EDIT THIS FILE MANUALLY`), re-export schema, declare `VersionFieldMetadata`, `TableMetadata`, `AdditionalMetadata`, and an empty `additionalMetadata` const. This file is normally generated by `forge-sql-orm-cli generate:model` — scaffolding it directly is fine for a new example so the user can edit before first CLI run.
- `src/migration/migrationCount.ts` — `export const MIGRATION_VERSION = 1;`
- `src/migration/migrationV1.ts` — enqueues CREATE TABLE statements matching the schema exactly (same column types, constraints, FK names, index names Drizzle would emit). For the `caching` template also enqueue the `ALTER TABLE ... ADD CONSTRAINT ... FOREIGN KEY` and `CREATE INDEX` statements as separate enqueues.
- `src/migration/index.ts` — standard wrapper that calls `v1(migrationRunner)` and returns it.
- `src/index.ts` — Forge resolver entry point. Structure:
  - Import `Resolver`, `ForgeSQL`, migration helpers (`applySchemaMigrations`, `dropSchemaMigrations`, `fetchSchemaWebTrigger`), entities, `drizzle-orm` operators as needed.
  - Template-specific resolvers (see below).
  - Export `handler`, `handlerMigration`, `dropMigrations`, `fetchMigrations`.

### Template-specific resolver set

**basic-crud** (`src/index.ts`): `create`, `delete`, `fetch` (with optional sort), `duplicate` (GROUP BY HAVING COUNT > 1). Use `new ForgeSQL({ logRawSqlQuery: true })` inline.

**caching**:
- Separate `src/utils/forgeSqlOrmUtils.ts` exporting `FORGE_SQL_ORM = new ForgeSQL({ logRawSqlQuery: true, cacheEntityName: "cache", cacheTTL: 180 })`.
- `manifest.yml` must include the `app.storage.entities` block with a `cache` entity (attributes: `sql: string`, `expiration: integer`, `data: string`; indexes on `sql` and `expiration`) and `permissions.scopes: - "storage:app"`.
- Resolvers: `webTriggers` (returns URLs), `fetch` with action switch (`cacheable`, `slow`, `outOfMemory`, `timeout`) wrapped in `executeWithMetadata`, `clearCache` (uses `modifyWithVersioningAndEvictCache().evictCacheEntities`), `insertUserOrOrder` (demonstrates `selectCacheableFrom`, `insertAndEvictCache`, `updateAndEvictCache`, `executeWithCacheContext`), `runPerformanceAnalyze`.
- Extra exports: `runSlowQuery` (uses `slowQuerySchedulerTrigger`), `handlerAsyncDegradation` (uses `printDegradationQueriesConsumer`).
- `manifest.yml` adds a `consumer` module for `degradationQueue` and a `scheduledTrigger` for `runSlowQueryFunc`.

**ai-vector**:
- Schema uses `vectorTiDBType("embedding", { dimension: N })` on the embedding column.
- `migrationV1` creates the table with `VECTOR(N) NOT NULL` and optionally enqueues seed INSERTs if the user wants starter data.
- Resolvers: `create`, `fetch` (returns all rows), `search` (takes a `vector: number[]`, computes `vecCosineDistance(...)` with `sql.raw` alias, orders asc). Import `vecCosineDistance` from `forge-sql-orm`.

### Optional: tests

If tests were requested, create `__tests__/resolvers.test.ts` inside the example using Vitest. Mock `@forge/sql` and `forge-sql-orm` at the module level. Cover one happy path per resolver. Do NOT try to run tests against a real database — that belongs in the root `__tests__/` integration suite, not the example.

## Consistency checks before finishing

Run mentally (or with Grep) after writing files:

1. Every column in `entities/schema.ts` appears in the corresponding `CREATE TABLE` SQL in `migrationV1.ts` with the same type.
2. Every resolver defined in `src/index.ts` has a matching `function` entry in `manifest.yml`, and vice versa.
3. The `caching` template's `cache` entity name in `manifest.yml` matches the `cacheEntityName` passed to `ForgeSQL(...)`.
4. `package.json` dependency versions match the chosen template's versions exactly.
5. `src/entities/index.ts` has the auto-generated warning comment — critical because `forge-sql-orm-cli generate:model` will overwrite it.

## After generation

Tell the user:
- The generated Forge `app.id` placeholder must be replaced with a real one from `forge register` / `forge create`.
- The `static/` frontend directory is NOT scaffolded; if they need a UI they should copy it from a neighboring example.
- To regenerate `src/entities/index.ts` against a real DB they can run `npm run models:create` from inside the example.
- To generate a migration diff from schema changes: `npm run migration:update`.

Do not create the `static/` directory. Do not create a `create-and-run-db.sh` unless the user asks — it's used only for local TiDB bootstrap and not every example has one (the `vector` example does not).
